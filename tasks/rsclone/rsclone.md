## Задание

RS Clone - финальный проект курса. Это командное задание, в ходе выполнения которого вам необходимо разработать клон игры или приложения. Команда сама решает какую игру или приложение копировать. Это может быть как точная, так и упрощённая копия, или ваше собственное приложение. 

## Тематика приложения
- при выборе тематики приложения или игры исходите из собственных интересов и возможностей. Оптимально, если игра или приложение, которые вы копируете, будут хорошо вам известны, вы будете знать особенности их работы, видеть возможные варианты их изменения и улучшения 
- выбирайте проект при работе над которым сможете проявить свои сильные стороны, реализовать свои возможности, получить максимально качественный результат

## Формирование команд
 - Оптимальное количество участников команды - два человека, состав команды может быть увеличен до четырёх участников, также допустимо выполнение задания в одиночку.
 - Команды формируются по желанию студентов.
 - Студенты могут попросить ментора курировать их проект, консультировать команду, проводить код-ревью. 
 - Для участия в проекте необходимо заполнить таблицу: вкладка [Проекты](https://docs.google.com/spreadsheets/d/1yD4LQ1SMYQeCOSVaxQkOEPm9szuth-rMlcJDw2EQ2zY/edit#gid=0) для уже сформировавшихся команд, вкладка [Разработчики](https://docs.google.com/spreadsheets/d/1yD4LQ1SMYQeCOSVaxQkOEPm9szuth-rMlcJDw2EQ2zY/edit#gid=1434189398) для студентов, которые ищут себе команду.

## Выполнение проекта
- распределите работу над приложением на недельные спринты, в результате выполнения первого спринта получите [MVP](https://ru.wikipedia.org/wiki/Минимально_жизнеспособный_продукт), в последующих спринтах улучшайте и совершенствуйте его
- Каждая команда ведет Trello board, в котором трекает процесс разработки 
- Каждая команда должна выполнить и выложить на youtube видеозапись демо игры или приложения.
- Каждая команда должна написать и выложить на medium статью о разработке приложения. При невозможности выложить статью на medium, допускается размещение статьи на другом ресурсе.
- После дедлайна лучшие 100 команд по результатам кросс-чека проводят презентацию своего проекта. 

## Требования к репозиторию
- для разработки приложения тим лид команды в своём аккаунте github создаёт приватный репозиторий, коллаборантами в который приглашает всех участников своей команды 
- особенности командной работы с репозиторием описаны в [Документации курса](https://docs.rs.school/#/final-task?id=Работа-с-репозиторием)
- название репозитория: **rsclone**, название ветки, в которой ведётся разработка - **develop**, ветка **master** пустая, содержит только README.md
- история коммитов должна отображать процесс разработки приложения. [Требования к коммитам](https://docs.rs.school/#/git-convention)
- демо-версия приложения размещается на `https://www.netlify.com/`, либо на другом подобном хостинге
- после окончания разработки или при наступлении дедлайна, создайте pull request из ветки develop в ветку master. [Требования к pull request](https://docs.rs.school/#/pull-request-review-process?id=Требования-к-pull-request-pr). Кроме перечисленных требований pull request должен содержать ссылку на статью на medium, ссылку на видео с обзором приложения на youtube, описание технологий применённых при создании приложения.
- репозиторий, в котором велась работа над проектом, после наступления дедлайна нужно сделать публичным.

## Технические требования
- работа приложения проверяется в браузере Google Chrome последней версии
- разрешается использовать jQuery только в качесте подключаемой зависимости для UI библиотек. Использование jQuery в основном коде приложения не допускается.
- можно использовать Angular / React / Vue, а также другие JS библиотеки (кроме JQuery) 
- можно использовать TypeScript
- можно использовать bootstrap, css фреймворки, html и css препроцессоры
- можно использовать Canvas, WebGL, SoundAPI и другие встроенные API браузера
- можно использовать графические движки Threejs, Phaser, Pixi, Babylon 
- рекомендуется создать и использовать бекэнд. Данная рекомендация связана с очень высоким спросом на фронтенд-разработчиков, знакомых хотя бы с основами node.js.

## Требования к оформлению приложения
- в футере приложения должны быть ссылки на гитхабы авторов приложения, год создания приложения, [логотип курса](https://rs.school/images/rs_school_js.svg) со [ссылкой на курс](https://rs.school/js/)

## Как сабмитить задание
- Каждый студент самостоятельно сабмитит в rs app ссылку на pull request и получает баллы по результатам кросс-чека.

- 160 баллов за статью

- Приложение и статья оцениваются в ходе кросс-чека, презентацию оценивает специально созданное жюри.
- Язык видео, статьи, презентации может быть английский, русский или белорусский

### Приложение
Максимальное количество баллов, которое можно получить за разработку приложения - 320. 
Каждая команда сама решает какие фичи будет реализовывать. 

##### UI:
* Есть возможность управление приложением/игрой с клавиатуры или наличие более пяти hot keys (20 баллов)
* Есть возможность переключения 2 и более языков (20 баллов)
* Есть хотя бы один модальный диалог (20 баллов)
* Реализован routing (без перезагрузки страницы) (приложение) (40 баллов)
* Возможность кастомизации приложения, настроек пользователя (30 баллов)
* Реализовано 3+ анимации (20 баллов)
* Приложение/игра выполнено в едином стиле (например, Bootstrap/Material UI/Ant design) (30 баллов)
* Приложение/игра отображает какую-либо статистику/графики/таблицы (20 баллов)
* Приложение/игра работает на телефоне/планшете/PC

##### Технический стек:
* Использован Canvas/WebGL (30 баллов)
* Работа с Audio API (20 баллов)
* Есть более двадцати Unit test (20 баллов)
* Использован webpack (20 баллов)
* Сохранение и загрузка чего-либо с использованием Local storage (20 баллов)
* Приложение/игра написанны на TypeScript (40 баллов)

##### Работа с кодом:
* Использован какой-либо data flow паттер MVC/MVVM/Redux/etc. (30 баллов)
* Использован eslint (20 баллов)
* Понятный, читаемый и «чистый» код (30 баллов)

##### Back-end:
* Использован RESTful API (30 баллов)
* Подключение и работа с БД (30 баллов)
* Аутентификация (20 баллов)
* Приложение отображает какую-либо статистику/графики/таблицы (30 баллов)
* Реализован nodejs и express, отдаёт корректные ответы, отдаёт HTTP ошибки с нормальными body, по которым можно понять, что произошло, пишет читаемые логи (40 балов)

### Презентация

Во время презентации представитель команды шарит свой экран и демонстрирует: 
1. Результат командной работы 
    - оформление и функционал игры или приложения, их особенности и возможности
2. Организацию командной работы
    - RACI матрицу содержащую список всех основных фич и процессов, роль участников команды в них. Рассказывает о вкладе каждого участника команды.
    - Kanban board. Рассказывает: 
      - как координировали деятельность команды
      - какие инструментами для организации совместной работы использовались
      - какие были сложности и как они были решены
3. Особенности технической реализации проекта и его репозиторий
     - readme.md содержащий описание библиотек, фреймворков, технологий, которые были задействованы в процессе разработки приложения, их преимущества и недостатки
     - историю коммитов
     - примеры PR. Рассказать как проходили кодревью, как обеспечивалось качество кода
     - технические сложности и проблемы с которым столкнулись в процессе создания приложения, как их решали
     - (по желанию) любые схемы, графики, диаграммы относящиеся к процессу разработки
4. Общее впечатление от презентации:  
    - эмоциональность, уверенность, убедительность, интересность, хороший темп подачи материала
    - развернутые ответы на вопросы жюри
